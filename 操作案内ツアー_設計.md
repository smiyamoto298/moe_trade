# 操作案内ツアー（オンボーディング）設計書

MoE_Trade の各ページに、操作方法を案内するステップ式ポップアップ（ツアー）を追加するための設計書。

- 表示方式: **初回自動表示 ＋ ヘルプボタン（「?」）による再表示**
- 見せ方: **ステップ式ツアー**（画面上の要素を順番にハイライトし「次へ」で進む）

---

## 1. 機能概要

### 1.1 基本動作

ツアーは「対象要素のハイライト ＋ 吹き出し（タイトル・本文・進捗）」を1ステップとし、複数ステップを順に表示する。

- **初回自動**: そのページに初めて来たユーザーには自動でステップ1から開始する。
- **ヘルプボタン**: ヘッダーに常設の「?」ボタンを置き、いつでも現在のページのツアーを再生できる。
- **既読管理**: ページ単位で「見た／見ていない」を `localStorage` に記録し、初回自動表示は1回だけにする。
- **操作**: 各ステップに「次へ」「戻る」「スキップ（×）」を用意。最終ステップは「完了」。
- **バージョニング**: 既読キーに版番号を付け（例 `_v1`）、案内内容を更新したら版を上げて再表示できるようにする。

### 1.2 既読キーの設計

```
tour_seen_<pageId>_<version>   // 例: tour_seen_listings_v1 = "1"
```

- ページ遷移時、対象 `pageId` のキーが無ければ自動開始 → 完了/スキップ時に `"1"` を書き込む。
- マイページに「操作案内をすべてリセット」ボタンを置き、全キーを削除して再表示できるようにする。

### 1.3 ステップのデータ構造

ツアー定義はページごとに配列で持つ。実装と内容を分離しておくと保守しやすい。

```ts
type TourStep = {
  target: string          // ハイライト対象のCSSセレクタ or data属性
  title: string           // 吹き出しの見出し
  body: string            // 説明文（1〜2文）
  placement?: 'top' | 'bottom' | 'left' | 'right'
}

type Tour = {
  pageId: string          // 既読管理のキー
  version: number
  steps: TourStep[]
}
```

- ハイライト対象には `data-tour="listings-tabs"` のような専用属性を各ページに振る（クラス名変更に強くするため）。
- 対象要素が存在しないステップ（権限やタブで非表示）は自動スキップする。

### 1.4 実装方針

既存スタック（React 18 + react-router + Tailwind、追加ライブラリ最小）に合わせ、以下のいずれか。

- **推奨: `react-joyride`** … React向けのツアー専用ライブラリ。ステップ配列を渡すだけでハイライト・吹き出し・進捗・スキップが揃う。スポットライト表示も標準対応で実装が速い。
- 代替: `driver.js`（フレームワーク非依存・軽量）または自前実装（依存を一切増やしたくない場合）。Tailwindで吹き出しを作り、対象要素の座標を計測してオーバーレイを重ねる。

共通コンポーネント案:

- `TourProvider` … 現在のページ・ステップ・既読状態を管理。`Header` の「?」ボタンと連携。
- `useTour(pageId)` … 各ページが自分のツアーを登録し、初回自動表示を制御するフック。
- ツアー定義は `src/tours/` にページ別ファイルで配置。

### 1.5 設計上の注意

- **モバイル対応**: 画面が狭いので吹き出しは要素の上下に自動配置。長文は避ける。
- **未ログイン/権限**: ログインが必要な操作は「ログインすると◯◯できます」と案内し、ボタンを押せない状態でも説明は出す。
- **動的要素**: 一覧が空・読み込み中だと対象が無い場合がある。ステップ開始前に要素の存在を確認し、無ければスキップ。
- **多重表示の防止**: 取引チャットやモーダルが開いている最中は自動ツアーを抑止する。

## 2. ページ別の案内内容

優先度の目安: **P1=最優先**（初回ユーザーが必ず通る／離脱しやすい）、P2=重要、P3=補助。

### 2.1 出品一覧（ListingsPage / `/listings`・`/skills`・`/assets`） — P1

サイトの入口。装備・スキル・資産の3モードを切り替える中核ページ。

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | 装備/スキル/資産タブ | 「ここで扱うアイテムの種類を切り替えられます」 |
| 2 | 検索・フィルタ（FilterPopup/StatRangeFilter） | 「名前・ステータス・価格帯で絞り込めます」 |
| 3 | 相場グラフ（PriceAnalytics） | 「過去の取引価格から相場の目安を確認できます」 |
| 4 | 出品カード | 「カードを押すと詳細・取引に進めます」 |
| 5 | 「出品する」「一括出品」ボタン | 「自分のアイテムを売りに出すならこちら（ログイン必要）」 |
| 6 | 買い求めへの導線 | 「探しているアイテムは『買い求め』で募集もできます」 |

### 2.2 出品詳細（ListingDetailPage / `/listings/:id`） — P1

取引が実際に始まる最重要ページ。

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | アイテム情報・ステータス | 「ステータスや価格、出品者の評価を確認できます」 |
| 2 | 相場分析モーダル | 「この価格が相場と比べて高いか安いかを確認できます」 |
| 3 | 取引リクエスト（TradeRequestPanel） | 「この出品者と取引したい場合はここから申し込み、チャットで交渉します」 |
| 4 | お気に入り／通報 | 「気になる出品の保存や、問題のある出品の通報ができます」 |

### 2.3 新規出品（NewListingPage / `/listings/new`） — P1

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | アイテム検索・選択 | 「まず出品するアイテムを検索して選びます」 |
| 2 | ステータス／価格入力 | 「ステータス値と希望価格を入力します。相場が参考表示されます」 |
| 3 | 認証の注意（VerifyEmailBanner） | 「出品にはメール／SMS認証が必要です」 |
| 4 | 出品ボタン | 「内容を確認して出品を確定します」 |

### 2.4 一括出品（BulkListingPage / `/listings/bulk`） — P2

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | 入力エリア／テンプレ | 「複数アイテムをまとめて入力・登録できます」 |
| 2 | 認証バナー | 「一括出品にも認証が必要です」 |
| 3 | 一括登録ボタン | 「入力内容をまとめて出品します。登録前にプレビューで確認を」 |

### 2.5 買い求め一覧／詳細／新規（BuyRequests* / `/buy-requests`系） — P2

「出品（売りたい）」と対になる「買い求め（買いたい募集）」の概念説明が要。

| ページ | # | 案内内容 |
|--------|---|----------|
| 一覧 | 1 | 「『買い求め』は欲しいアイテムを募集する掲示です。出品の逆向きです」 |
| 一覧 | 2 | 「自分が持っているアイテムなら、募集に応じて売ることができます」 |
| 詳細 | 1 | 「条件を確認し、応じる場合はチャットで交渉します」 |
| 新規 | 1 | 「欲しいアイテムと希望条件・予算を入力して募集を出します」 |

### 2.6 マイページ（MyPage / `/mypage`） — P1

4タブ（出品中／購入中／買い求め／売り対応）＋取引チャットがあり、最も複雑。

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | タブ全体 | 「自分の取引を4つの立場で確認できます」 |
| 2 | 出品中タブ | 「自分が売りに出しているアイテムと、来た取引リクエスト」 |
| 3 | 購入中タブ | 「自分が申し込んだ取引の進行状況」 |
| 4 | 買い求めタブ | 「自分が出した募集と、応じてくれた相手」 |
| 5 | 売り対応タブ | 「他人の募集に応じている取引」 |
| 6 | 取引チャット／候補選択（CandidateSelectModal） | 「相手とやり取りし、取引相手を選んで取引を進めます」 |
| 7 | 取引完了・評価 | 「取引が終わったら完了にして、相手を評価します」 |

### 2.7 運営掲示板（BoardPage / BoardThreadPage / `/board`系） — P3

| # | 対象 | 案内内容 |
|---|------|----------|
| 1 | スレッド一覧／フィルタ | 「運営からのお知らせや相談用の掲示板です」 |
| 2 | 投稿・返信 | 「スレッドを開いて閲覧・返信できます」 |

### 2.8 認証系（Login / Register / ForgotPassword / ResetPassword） — P2

フォームが単純なのでツアーより**1ステップの補足**で十分。

| ページ | 案内内容 |
|--------|----------|
| 登録 | 「登録後にメール／SMS認証を行うと出品・取引ができます」 |
| ログイン | 「パスワードを忘れた場合は下のリンクから再設定できます」 |

### 2.9 管理系（admin/* ） — P3

権限者のみ。ツアーは作らず、各操作ボタンに**ツールチップ**を添える程度で十分。

| ページ | 補足 |
|--------|------|
| アイテム管理 | 「相場登録・アイテム編集の権限者向け画面」 |
| ユーザー管理／お知らせ | admin専用。各操作にホバー説明 |

### 2.10 全ページ共通要素

- **ヘッダーナビ**: 初回のみ「上のメニューで出品一覧・買い求め・マイページを行き来できます」を最初に1ステップ。
- **認証バナー（VerifyEmailBanner）**: 未認証ユーザーに「認証すると出品・取引が解放されます」を案内。

---

## 3. 実装の進め方（提案）

1. **共通基盤** … `react-joyride` 導入、`TourProvider`／`useTour`／既読管理、ヘッダーに「?」ボタンを追加。
2. **P1ページから着手** … 出品一覧 → 新規出品 → 出品詳細 → マイページ。最も離脱しやすい導線を先にカバー。
3. **P2／P3を順次追加** … 一括出品・買い求め・認証 → 掲示板・管理。
4. **`data-tour` 属性の付与** … 各ページのハイライト対象に専用属性を振る（クラス変更に強くする）。
5. **検証** … 初回自動表示／既読化／?ボタン再表示／モバイル表示／対象なしステップのスキップを確認。

---

## 4. 確認したいこと

- ツアーの**文言トーン**（丁寧な敬体 / フランクな口調）はどちらが好みか。
- 「?」ボタンの**設置場所**（ヘッダー固定 / 画面右下のフローティング）。
- まずは**P1の4ページだけ実装**して試すか、全ページ一気に作るか。
